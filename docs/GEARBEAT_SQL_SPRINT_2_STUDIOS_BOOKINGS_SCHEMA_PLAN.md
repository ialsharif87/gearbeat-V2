## 1. PURPOSE OF SQL SPRINT 2
This phase focuses on the primary vertical of GearBeat: **Studios**. It defines how studios are described, how their availability is managed, and how the transactional booking lifecycle is tracked from request to completion.

## 2. DEPENDENCY ON SQL SPRINT 1
This schema directly inherits foundations from SQL Sprint 1:
- **`partner_accounts`**: Every studio must be associated with a `partner_account_id` (representing the business entity).
- **`profiles`**: Used to identify customers making bookings and studio owners.
- **`role_definitions`**: Policies check for `studio_owner` and `customer` roles.

## 3. STUDIO DOMAIN REQUIREMENTS
- **Discovery**: Rich metadata (media, equipment, rooms) to drive the high-fidelity search and profile experience.
- **Availability**: Granular control over opening hours (rules) and specific closures (blackout dates).
- **Pricing**: Flexible base rates with the ability to override based on date or room type.

## 4. BOOKING DOMAIN REQUIREMENTS
- **Request Flow**: Support for "created" states that wait for payment confirmation before becoming a confirmed booking.
- **Traceability**: `booking_number` generation and `booking_status_history` for full auditability of state changes.
- **Post-Booking**: Reviews and ratings tied to specific booking IDs to prevent fake review spam.

## 5. BOOKING LIFECYCLE STATES
1. `created`: Initial request generated by customer.
2. `pending_payment`: Awaiting payment confirmation (manual or future provider).
3. `confirmed`: Payment verified; studio slot reserved.
4. `completed`: Booking time has passed.
5. `cancelled`: Cancelled by user or studio; triggers refund workflow if applicable.

## 6. PAYMENT BOUNDARIES
- **Manual Payment (Current)**: Tracks `manual` as a provider with local references.
- **Future Tap Payment**: Includes placeholders for `external_reference_id` and `payment_intent_token` to allow seamless transition to live gateways.
- **Security**: No credit card data is stored.

## 7. RLS DIRECTION (DRAFT)
- **`studios`**: 
  - `SELECT`: Public (if published).
  - `INSERT/UPDATE`: Restricted to `studio_owner` of the associated `partner_account`.
- **`studio_bookings`**:
  - `SELECT`: Restricted to the `customer` (user_id) or the `studio_owner`.
  - `INSERT`: Restricted to `customer` role.
  - `UPDATE`: Restricted to `admin` or specific `studio_owner` actions (e.g., mark as completed).

- **Data Integrity**: Transitioning from a flat `images` array to a `studio_media` table requires careful data mapping.
