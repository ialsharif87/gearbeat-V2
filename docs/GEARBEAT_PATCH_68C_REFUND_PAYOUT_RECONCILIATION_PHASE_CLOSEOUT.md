# GEARBEAT PATCH 68C — REFUND / PAYOUT / RECONCILIATION + PHASE 68 CLOSEOUT

## 1. Overview
This is a **documentation and planning-only** patch for GearBeat V2. It finalizes the financial operations strategy by defining the refund, payout, and reconciliation lifecycles. This document also serves as the formal closeout for **Phase 68: Payment Pre-Activation Foundation**.

**Policy:** Documentation-only. No mutations to backend, API, database, or payment logic.

---

## 2. Refund Lifecycle Planning

### A. Studio Bookings
- **Trigger:** Customer cancellation (within policy) or Studio technical failure.
- **Process:** `payment_refunds` record created -> Admin review -> API call to Tap Refund API -> Status update to `refunded`.
- **Integrity:** Refunds must be linked to the original `payment_transaction_id`.

### B. Marketplace Orders
- **Trigger:** Product return (verified) or out-of-stock cancellation.
- **Process:** Return received -> Quality check -> Admin trigger refund -> Partial or full amount released via Tap.
- **Constraint:** Shipping fees are non-refundable unless the product was defective.

---

## 3. Payout Planning

### A. Studio Owners & Marketplace Vendors
- **Frequency:** Bi-weekly or monthly settlements.
- **Eligibility:** Funds move from `pending_payout` to `available_payout` only after the 14-day marketplace return window or 24 hours post-booking.
- **Mechanism:** GearBeat settles via manual bank transfer (SAR) to the IBAN on record until automated payout APIs are active.

### B. instructors & Service Providers
- **Process:** Payouts triggered upon verified completion of the service/course.

---

## 4. Reconciliation Planning
To ensure transaction integrity, the system will compare three data sources:
1.  **GearBeat Internal Ledger:** `payment_transactions` and `bookings`/`marketplace_orders`.
2.  **Tap Payments Dashboard:** Verified charge and settlement reports.
3.  **Bank Statements:** Final settlement of funds into the GearBeat corporate account.

**Anomaly Rule:** Any mismatch between GearBeat status and Tap Webhook status triggers an automatic "Admin Review Required" alert.

---

## 5. Tap Future Boundaries (Phase 69 Implementation)
- **Session Source of Truth:** `checkout_payment_sessions` must exist before a Tap charge is created.
- **Webhook Source of Truth:** Only verified webhooks can update an order/booking to `paid`.
- **Idempotency:** Tap `idempotency-key` used for all capture and refund requests.
- **Reporting:** Daily reconciliation reports generated by comparing `provider_reference` against internal IDs.
